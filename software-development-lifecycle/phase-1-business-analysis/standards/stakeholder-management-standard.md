# Stakeholder Management Standard

| Field | Value |
|-------|-------|
| **Standard ID** | STD-P1-003 |
| **Compliance Level** | Mandatory |
| **Enforced By** | Product Owner |
| **Verification Method** | Phase 1 Gate Review |
| **Last Verified** | 2026-03-04 |

---

## SCALING GATE

> **IF** small project **THEN** Stakeholder register with minimum 3 categories identified; Power/Interest classification optional; communication plan limited to key stakeholders
> **IF** medium project **THEN** Standard compliance; all 6 stakeholder categories evaluated, Power/Interest classification required, communication plan per quadrant
> **IF** large project **THEN** Full compliance; all 6 categories with named individuals, Power/Interest classification with documented rationale, detailed communication plan per stakeholder, formal update triggers enforced

---

## COMPLIANCE REQUIREMENTS

### Stakeholder Identification Rules

| # | Rule | Applies To | Verification | Violation Response |
|---|------|-----------|-------------|-------------------|
| 1 | Stakeholder register MUST be created and maintained | All projects | [ ] Stakeholder register exists | Reject at gate review; return for revision |
| 2 | Each stakeholder entry MUST include name, role, organization, and contact information | Medium/Large | [ ] All stakeholder entries have required fields | Return for revision |
| 3 | Each stakeholder entry MUST include name and role at minimum | Small | [ ] All stakeholder entries have name and role | Return for revision |
| 4 | Stakeholder identification MUST involve at least the Sponsor and Product Owner | All projects | [ ] Sponsor and PO participated in identification | Return for revision |
| 5 | Stakeholder register MUST be reviewed for completeness by Project Sponsor | Medium/Large | [ ] Sponsor review recorded | Cannot proceed to Phase 1 gate |

### Minimum Stakeholder Categories

| # | Rule | Applies To | Verification | Violation Response |
|---|------|-----------|-------------|-------------------|
| 6 | Internal Executive stakeholders MUST be identified (decision-makers, senior leadership) | All projects | [ ] Internal Executive category evaluated | Reject at gate review; return for revision |
| 7 | Internal Operational stakeholders MUST be identified (teams affected by delivery, support staff, operations) | All projects | [ ] Internal Operational category evaluated | Reject at gate review; return for revision |
| 8 | End User stakeholders MUST be identified (direct users of the system or product) | All projects | [ ] End User category evaluated | Reject at gate review; return for revision |
| 9 | External Customer stakeholders MUST be identified (paying customers, clients, partners receiving output) | Medium/Large | [ ] External Customer category evaluated or documented as not applicable | Return for revision |
| 10 | Regulatory stakeholders MUST be identified (compliance bodies, auditors, legal) | Medium/Large | [ ] Regulatory category evaluated or documented as not applicable | Return for revision |
| 11 | Partner stakeholders MUST be identified (vendors, suppliers, third-party integrators) | Medium/Large | [ ] Partner category evaluated or documented as not applicable | Return for revision |
| 12 | Each category MUST have at least 1 named stakeholder OR explicit documentation that the category is not applicable with justification | Medium/Large | [ ] Each category populated or justified as N/A | Return for revision |

### Power/Interest Classification Rules

| # | Rule | Applies To | Verification | Violation Response |
|---|------|-----------|-------------|-------------------|
| 13 | Each stakeholder MUST be classified on a Power/Interest grid | Medium/Large | [ ] All stakeholders classified on Power/Interest grid | Reject at gate review; return for revision |
| 14 | Power MUST be rated as High or Low based on authority to influence project decisions | Medium/Large | [ ] Power rating assigned to each stakeholder | Return for revision |
| 15 | Interest MUST be rated as High or Low based on degree of concern with project outcomes | Medium/Large | [ ] Interest rating assigned to each stakeholder | Return for revision |
| 16 | Classification MUST result in one of four quadrants: Manage Closely (High Power/High Interest), Keep Satisfied (High Power/Low Interest), Keep Informed (Low Power/High Interest), Monitor (Low Power/Low Interest) | Medium/Large | [ ] Each stakeholder assigned to a quadrant | Return for revision |
| 17 | Classification rationale MUST be documented for each stakeholder | Large | [ ] Rationale recorded per stakeholder | Return for revision |
| 18 | Classification MUST be reviewed and validated by Product Owner | Medium/Large | [ ] PO validation recorded | Return for revision |

### Communication Plan Requirements

| # | Rule | Applies To | Verification | Violation Response |
|---|------|-----------|-------------|-------------------|
| 19 | Communication plan MUST be defined for key stakeholders | All projects | [ ] Communication plan exists | Reject at gate review; return for revision |
| 20 | Manage Closely (High Power/High Interest): MUST have weekly or more frequent communication with direct engagement method | Medium/Large | [ ] Manage Closely stakeholders have weekly+ cadence and direct engagement | Return for revision |
| 21 | Keep Satisfied (High Power/Low Interest): MUST have at minimum monthly communication with status report method | Medium/Large | [ ] Keep Satisfied stakeholders have monthly+ cadence and status reports | Return for revision |
| 22 | Keep Informed (Low Power/High Interest): MUST have at minimum bi-weekly communication with newsletter or group update method | Medium/Large | [ ] Keep Informed stakeholders have bi-weekly+ cadence and group updates | Return for revision |
| 23 | Monitor (Low Power/Low Interest): MUST have at minimum quarterly communication with general broadcast method | Medium/Large | [ ] Monitor stakeholders have quarterly+ cadence | Return for revision |
| 24 | Each communication plan entry MUST include: stakeholder/group, method, frequency, content type, responsible party | Medium/Large | [ ] All communication plan entries have required fields | Return for revision |
| 25 | Each communication plan entry MUST include: stakeholder/group, method, and frequency at minimum | Small | [ ] Small project communication plan has minimum fields | Return for revision |
| 26 | Communication plan MUST designate a single responsible owner for each stakeholder or stakeholder group | Large | [ ] Communication owner assigned per entry | Return for revision |
| 27 | Escalation path MUST be defined for unresponsive or disengaged high-power stakeholders | Large | [ ] Escalation path documented | Return for revision |

### Stakeholder Register Update Triggers

| # | Rule | Applies To | Verification | Violation Response |
|---|------|-----------|-------------|-------------------|
| 28 | Stakeholder register MUST be updated when a new stakeholder is identified | All projects | [ ] Process for adding new stakeholders documented | Return for revision |
| 29 | Stakeholder register MUST be updated when a stakeholder's role or authority changes | Medium/Large | [ ] Role/authority change trigger documented | Return for revision |
| 30 | Stakeholder register MUST be updated when a stakeholder leaves the organization or project | Medium/Large | [ ] Departure trigger documented | Return for revision |
| 31 | Stakeholder register MUST be updated when project scope changes materially | Medium/Large | [ ] Scope change trigger documented | Return for revision |
| 32 | Stakeholder register MUST be updated at each phase gate transition | Medium/Large | [ ] Phase gate trigger documented | Return for revision |
| 33 | Stakeholder register MUST be reviewed at minimum monthly for accuracy | Large | [ ] Monthly review cadence documented | Return for revision |
| 34 | All updates MUST include date, nature of change, and who authorized the change | Large | [ ] Update audit trail requirements documented | Return for revision |
